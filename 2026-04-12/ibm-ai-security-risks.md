# 从 Claude Code 泄漏到 Team PCP：AI 时代的安全边界，已经从漏洞转向“信任链”

> 来源视频：IBM Technology / *Claude Mythos, Project Glasswing and AI cybersecurity risks*  
> 视频链接：https://www.youtube.com/watch?v=DGnHbF-KtKU  
> 发布时间：2026-04-10

## 一句话结论

这期 IBM 节目的核心判断很明确：**AI 安全问题，已经不再只是“模型会不会出错”，而是整个软件供应链、身份体系和代理式 AI（Agentic AI）信任链会不会被绕开。**

换句话说，真正危险的不是某一个模型泄漏了源码，而是这件事让所有人都看见：当 AI 工具深度接入 npm、CI/CD、云凭证、MCP 接口和企业工作流之后，攻击者要打的就不再是一个点，而是一整条链。

这也是为什么节目从 Claude Code 源码泄漏，一路谈到 Team PCP 的凭证滥用攻击、再谈到企业如何应对——因为这些新闻看起来分散，本质上其实是一回事：**攻击者正在从“找漏洞”转向“劫持信任”。**

---

## 1. Claude Code 源码泄漏，真正暴露的不是 Anthropic，而是整个 AI 供应链

节目先聊的是最近很轰动的一件事：Anthropic 不小心在 npm 发布了带有 Claude Code 源码的版本，导致完整源代码短暂暴露。随后，攻击者马上在 GitHub 上投放假仓库，伪装成这份源码的 fork，并借此传播恶意软件，例如信息窃取木马。

表面上看，这是一次“源码泄漏事故”。但 IBM 几位嘉宾的判断都很一致：**这不是单纯的 Claude 问题，而是一个 AI 时代的软件供应链问题。**

节目里提到，npm 生态长期存在软肋：

- typosquatting（拼写相似包投毒）
- dependency confusion（依赖混淆）
- 恶意包污染工作流
- 通过包管理器污染构建链路

所以这次事件最值得警惕的地方，不是“Claude 的代码被人看到了”，而是攻击者借机看清了代理式 AI 工具的内部结构、权限链和未来能力方向。节目甚至提到，源码中可能能让人窥见未来功能形态，比如更强的离线能力、自动化能力等，这相当于**直接降低了攻击研究成本**。

IBM 嘉宾引用的一个观点很到位：这次泄漏不是某个产品的局部失误，而是把 Agentic AI 的系统性脆弱面暴露到了聚光灯下。以前攻击者想研究这类工具，需要逆向、黑盒测试、慢慢猜。现在不一样了——门槛被直接打穿了。

### 我的看法

这件事最值得企业记住的一句，不是“别泄漏源码”，而是：

**AI 工具一旦进入生产环境，它就不再只是一个开发工具，而是一个有权限、有上下文、有执行能力的供应链节点。**

它能读代码、调接口、进 CI/CD、调用云服务、拿环境变量、连接 MCP。你把它放在哪，它就会天然继承那里的攻击面。

所以，今天讨论 Claude Code，明天就会轮到任何企业内部的 coding agent、ops agent、research agent。

## 2. AI coding 工具的源码泄漏，为什么比普通源码泄漏更危险？

节目里还有一个很现实的问题：**AI 编码工具的源码泄漏，会不会比普通软件源码泄漏更严重？**

答案基本是“会，而且通常更严重”。

原因不只是它“会写代码”，而在于它通常具备以下几个特点：

1. **与敏感系统深度集成**  
   这类工具往往嵌在开发环境、代码仓库、CI/CD、包管理器和云部署流程里。

2. **拥有自动化执行能力**  
   它不仅能“建议”，还可能能“执行”。一旦被改造，攻击者等于拿到了一个高权限自动化打工人。

3. **防护逻辑本身可能被研究和绕过**  
   节目里提到一个关键点：源码泄漏可能让攻击者更容易理解工具原本的 guardrails（护栏）是怎么设计的，从而更快地拆护栏。

4. **攻击方式更像“复制 + 武器化”**  
   短期内最常见的风险，不一定是高级零日，而是大量用户去下载所谓的“破解版”“泄漏版”“开源 fork 版”，结果中招木马。这个路径老派，但特别有效。

IBM 的嘉宾有个判断我挺认同：短期风险是 Trojanized 假版本横行，长期风险则是**攻击者把这类 agent 重新包装成恶意自动化平台**。

这意味着什么？意味着过去你担心的是黑客手动打你；以后你担心的是黑客拿着现成的 AI 工具，24/7 自动打你。

## 3. 安全重心正在变化：从“漏洞”转向“信任链”

节目里我认为最值得记下来的观点，是这一句：

> 我们正在从“漏洞时代”，走向“信任链被颠覆的时代”。

这句话非常准。

过去做安全，大家的思路是：

- 找 CVE
- 打补丁
- 升级版本
- 配 WAF
- 做扫描

这些当然还重要，但对 Agentic AI 来说已经不够了。因为很多时候，攻击者不是“攻破”系统，而是**借用系统原本就信任的链路**：

- 从 npm 包进来
- 从 CI token 进来
- 从 GitHub Actions 凭证进来
- 从云环境凭证进来
- 从 MCP 工具接口进来
- 再横向进入你的工作流

这就是为什么传统“补漏洞”思路越来越吃力。不是因为漏洞没了，而是因为**合法链路本身成了攻击载体**。

对企业来说，这会逼着安全策略做几个转向：

- 不只看代码是否安全，也要看依赖来源是否可信
- 不只看系统有无漏洞，也要看代理是否被过度授权
- 不只看模型输出，也要看模型能调用哪些工具
- 不只看账户密码，也要看整条身份与权限传播链

一句话：**AI 时代最贵的资产不只是数据，而是“可被信任地执行动作的能力”。**

## 4. Team PCP 事件：攻击者真正抢的不是服务器，而是“身份”

节目后半段转向另一个案例：Team PCP 的持续入侵活动。

这波攻击的套路是：先从开源项目、扫描器、构建流程等环节窃取凭证，再把这些凭证用于云和 SaaS 环境横向移动，甚至波及欧洲委员会的云环境，并连带影响多个机构。

节目里反复强调一个细节：**很多时候不需要很多凭证，一个就够。**

这听着很扎心，但很真实。

例如某些场景里，防守方明明已经意识到凭证泄漏，并开始轮换，但只漏掉一个实例、一个遗留 token、一个没清理干净的 secret，攻击者就能钻进去。

IBM 嘉宾给出的核心判断是：

**身份（Identity）已经成了新的边界，甚至是最后一道边界。**

这句话我也完全赞同。因为在云原生和代理式 AI 世界里，传统边界越来越模糊：

- 内网和外网没那么分明了
- 应用和平台的边界也被 API 打平了
- 人工操作和自动化流程混在一起
- Bot、Agent、Workflow 都在“像用户一样”工作

最后真正能区分“你是谁、你能干什么、你现在凭什么能干”的，只剩身份体系。

所以今天很多攻击事件，表面是供应链攻击，实质上打的是身份系统：

- 谁有 token
- token 有多长寿命
- token 能访问哪些资源
- 凭证轮换是否彻底
- 权限是否最小化
- 是否存在僵尸权限

企业以前把 IAM 当治理问题，现在最好把它当**生存问题**。

## 5. 攻击者为什么越来越快？因为他们在和“凭证过期”赛跑

节目里还有一个判断特别现实：Team PCP 这类攻击者之所以动作又快又猛，原因之一可能是他们知道防守方会开始上短期凭证、轮换策略和更严格的审计，因此他们必须抢时间。

这很像“偷到钥匙后赶紧去试门，趁屋主还没换锁”。

这也是为什么现在很多安全事故会呈现一种很刺眼的风格：

- 不够隐蔽
- 不够优雅
- 但非常快、非常凶、非常有效

从防守角度看，这会带来两个现实后果：

### 第一，检测和响应窗口正在缩短

以前你可能还有几天时间去发现异常。现在，攻击者几小时内就能完成：

- 窃取凭证
- 登录 SaaS
- 拉数据
- 横向扩散
- 建立持续访问

### 第二，“等补丁”“等验证”“等审批”的节奏太慢

节目里有人提到，现在甚至连“第一时间升级最新版本”都未必永远安全，因为最新版本可能还没经过充分验证。于是很多团队只能采用一种很保守但现实的策略：**保持在相对稳定、经过验证的安全版本后面一小步。**

这虽然不性感，但很像成熟组织会做的事——别做第一个吃螃蟹的人，尤其别在核心生产链路里做。

## 6. 企业该怎么应对？我认为至少做这 6 件事

结合节目内容，如果把它翻译成企业今天就能执行的动作，我会建议从下面六件事开始。

### 1）把 AI 工具视为高权限基础设施，而不是普通插件

别再把 coding agent、自动化 agent 当“聪明一点的 IDE 插件”。它们本质上已经是能够执行动作的半自动操作节点。

要像管理 CI/CD、云账户、跳板机一样管理它们。

### 2）全面收紧供应链信任

至少要做到：

- 锁定依赖来源
- 验证包名和发布者
- 做制品签名校验
- 隔离构建环境
- 对引入新包和新工具建立审批与审计

### 3）默认所有 agent 都可能踩中 lookalike/typosquatting 陷阱

人类有时能看出拼写别扭，但 agent 不一定。任何自动装包、自动执行 shell、自动调用脚本的流程，都应该对“相似包名”和“异常来源”做额外拦截。

### 4）把凭证寿命压到最短

短期凭证不是万能药，但绝对比长期密钥强。能用临时凭证的地方，就别用永久凭证。

### 5）把权限做细，而不是做大

最小权限这四个字大家都会说，但 agent 时代要执行得更狠。因为一旦代理被滥用，它继承到的每一层权限都会变成攻击者的工作半径。

### 6）在升级与采用新 AI 能力时，先实验室、后生产

节目里几位嘉宾都偏向一个保守结论：**先在实验室里测透，再上生产。**

这听起来老派，但很对。AI 工具的变化速度太快，如果你每个新功能都第一时间在生产里开全权限，基本等于邀请问题上桌。

## 7. 这期节目真正提醒我们的，不是“AI 不安全”，而是“AI 正在继承一切旧风险并放大它们”

很多人看到这类新闻，第一反应是：AI 真危险。

我反而觉得，更准确的说法是：

**AI 并没有发明全新的所有风险，它只是把原本就存在的供应链风险、身份风险、自动化风险和权限风险，放大到了一个前所未有的级别。**

以前依赖投毒，影响的是开发机。现在可能影响整个智能代理链。

以前凭证泄漏，影响的是单个账户。现在可能影响整条自动化工作流。

以前脚本执行错误，影响的是一台机器。现在 agent 可能把错误高效复制到十个系统。

这才是真正麻烦的地方。

所以这期节目最重要的价值，不在于讨论某个新闻八卦，而在于逼着企业承认一个现实：

> **AI 安全不是模型安全的子集，而是现代软件供应链安全、身份安全和自动化治理的总和。**

如果你只盯着 prompt injection，却不看 npm、token、CI/CD、MCP、权限边界，那基本是在用望远镜看脚下的坑。

---

## 结语

这期 IBM 节目讲了两件新闻：Claude Code 源码泄漏、Team PCP 凭证攻击。但真正串起来看，核心只有一句：

**攻击者已经学会不去硬砸墙，而是顺着你信任的门进来。**

而在 AI 时代，这扇门不只属于人，也属于包管理器、构建流水线、API、代理和自动化系统。

如果企业现在还把 AI 安全理解成“给模型加点护栏”，那就太天真了。真正该加护栏的，是整个执行链路。

守不住信任链，模型再聪明也只是给攻击者打工。
