# 从Anthropic到OpenAI：Agentic Harness Engineering 正在重塑AI开发

## 什么是 Harness Engineering？

在2026年上半年，AI工程圈升温最快的概念莫过于 **Harness Engineering**。如果你还停留在“怎么写一条更好的 prompt”层面做 Agent，那么你可能已经落后了。决定 Agent 上限的，不再是你喂了什么 prompt 或选了什么模型，而是你给模型搭了一套什么样的运行系统——也就是 Harness。

Mitchell Hashimoto 在2026年初给出了一个极简洁的定义：“每当你发现 Agent 犯了一个错误，你就花时间工程化一个解决方案，让它永远不再犯同样的错”。Harness Engineering 不是教模型怎么回答，而是设计模型怎么工作：任务拆解、上下文管理、工具编排、权限设定、状态交接、验证恢复等。

## 概念演进：Prompt -> Context -> Harness

1. **Prompt Engineering (2022-2024)**: 关注“问什么”，本质是单轮文本优化。
2. **Context Engineering (2025)**: 关注“给模型看什么”，包含RAG、记忆注入、对话历史管理。
3. **Harness Engineering (2026)**: 关注“模型在什么机制里干活，如何确保它把活干成”。它不仅管看什么，还管工具、权限、状态验证、失败重试及人类接管。

## 四大驱动力

1. **模型能力抬高**：系统设计成为主要差异来源。
2. **长时任务缺陷**：长任务暴露出裸模型的系统性缺陷（如上下文迷失），需要系统级治理。
3. **概率累积衰减**：多步流水线中每步95%的成功率，串联20步后仅剩36%，需靠系统级重试和检查点恢复。
4. **模型商品化**：旧护城河是模型质量，新护城河是 Harness 质量。

## 巨头实践解析

### Anthropic：双/三 Agent 架构与外部记忆
- **双 Agent 架构**：初始化 Agent 建立环境与需求清单，编码 Agent 负责后续执行。
- **外部记忆**：进度文件、Git历史、结构化需求跨会话持久化。
- **三 Agent 演进**：Planner（计划）、Generator（生成）、Evaluator（评估）。Anthropic 发现模型自评估存在缺陷，独立评估器是最佳解法。

### OpenAI：百万行代码实验
- 7人团队5个月内用 Codex 生成百万行代码，零手写代码。
- **三大支柱**：
  1. 上下文工程（增强代码知识库与动态访问）。
  2. 架构约束（自定义linter与结构测试）。
  3. 垃圾回收（后台扫描文档不一致与架构违规）。

### Google DeepMind：Aletheia
- 面向数学研究的自主 Agent，采用三组件架构：Generator、Verifier、Reviser。
- 与 Anthropic 殊途同归，证明了 **“生成-评估分离”** 正在成为行业共识。

### Vercel：少即是多
- 移除 80% 的工具后，Agent 表现反而更好。
- 结论：约束 Agent 的解决空间，反而能提升其表现。更多的约束比更少的护栏更能带来 AI 自主性。

## 成熟 Harness 的六大核心模块
1. **上下文工程与知识管理**：项目指令文件、上下文隔离与压缩。
2. **工具编排与权限设计**：精简工具集、MCP集成、沙箱隔离。
3. **验证机制与硬约束**：确定性约束、生成-评估分离、自动审查循环。
4. **状态管理与记忆持续性**：外部化记忆、增量Git提交、检查点恢复。
5. **可观测性与反馈闭环**：执行追踪、异常检测、反馈归因。
6. **人类接管与生命周期管理**：关键决策点暂停、升级路径与重试。

## 总结
Prompt 解决“怎么说”，Context 解决“给模型看什么”，Harness 解决“怎么确保活被干成”。Agent 不难，Harness 才难。
