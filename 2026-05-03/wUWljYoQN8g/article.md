# 40 分钟讲透 Context Engineering：Ravi Mehta 这场演示，为什么值得每个做 AI 产品的人反复看

- 视频标题：Everything You Need to Know About Context Engineering in 40 Minutes | Ravi Mehta
- YouTube：<https://youtu.be/wUWljYoQN8g>
- 时长：38 分 49 秒
- 嘉宾：Ravi Mehta（前 Tinder CPO，AI 产品顾问）
- 主持人：Peter Yang

## 先说结论

如果你现在还把 AI 原型开发理解成“写一个 prompt，丢给 Claude / Cursor / v0，然后等它吐个页面出来”，那你大概率正在制造 Ravi 口中的 **AI slop**。

这期视频最有价值的地方，不是又发明了一个新名词，而是把一个很多人隐约感受到、但说不清的问题讲透了：

> **问题从来不是 prompt 写得不够花，而是你给模型的上下文，远远不够完整。**

Ravi 给出的替代框架叫 **Context Engineering**。他的定义很干脆：

> Context engineering is designing and building systems that provide an AI model with the right information and tools to accomplish the task.

翻成人话就是：

**你不是在和模型聊天，而是在为模型设计工作环境。**

这个视角一换，很多事就 suddenly clicks：

- 为什么很多 AI demo 看起来“能跑，但很糙”；
- 为什么同一个模型，有人能做出像样产品，有人只能做出塑料感原型；
- 为什么“会不会写 prompt”很快会变成次要能力；
- 为什么真正的门槛，开始从 prompting 转向 **系统化组织上下文**。

我看完这期后的核心判断是：

1. **Prompt engineering 没死，但它降级了。**它只是 context engineering 里的一个子模块。
2. **高质量 AI 原型，本质上是多层上下文共同工作的结果。**
3. **谁更会组织 functional / visual / data 这三层上下文，谁就更容易做出不像 AI slop 的产品。**
4. **未来产品经理、设计师、工程师的共同语言，会越来越像“上下文架构”。**

下面我把这期内容拆开讲。

---

## 一、为什么很多 AI 原型总有一股“塑料味”

Ravi 一开场就点破了一个常见误区：

很多人做 AI prototype 的方式，依然停留在“写一个 quick prompt 或 mini spec，然后期待工具直接给出高保真结果”。

问题在于，这种做法默认模型会自动脑补你没说出来的东西——但现实通常相反：

- 你没定义产品该做什么，它就会瞎补；
- 你没说明界面应该长什么样，它就会套模板；
- 你没给真实数据结构，它就会塞 dummy data；
- 你没准备外部工具，它就会停在一个“假装可用”的状态。

于是你得到的结果往往是：

- 页面看着像那么回事；
- 但一细看就不自然；
- 信息结构松散；
- 视觉语言不统一；
- 数据是假的；
- 用户一上手就出戏。

这就是 Ravi 反复提到的那种感觉：**it feels like AI slop**。

我很认同这一点。现在不少人讨论“AI 做产品到底行不行”，其实讨论对象都不一样。

有些人用 AI 的方式是：

- 丢一句需求；
- 让模型自由发挥；
- 然后拿一个半成品出来评判能力。

这有点像把一个实习生扔进会议室，只给一句“你先做个方案”，既不给背景、不给素材、不给规范、不给数据，还要求他 10 分钟交一版能给客户看的东西。最后你说“这人能力一般”。

不是他不行，是你组织工作的方式太糙了。

## 二、Context Engineering 到底是什么

Ravi 对这个概念的解释非常实用。

传统的 prompt engineering，更像是在优化你对模型“说什么”。

而 context engineering，更像是在设计：

- 模型能看到什么信息；
- 这些信息以什么结构出现；
- 模型能调用哪些工具；
- 输入之间如何彼此配合；
- 最终输出怎样更接近真实产品工作流。

这是从“写一句聪明的话”到“搭一个能工作的系统”的转变。

这也是为什么 Ravi 说，context engineering 代表着一种更准确的 framing：

> 你不是在进行一场对话，而是在为模型完成任务所需的信息与工具做整体设计。

这句话其实很重要，因为它把 AI 产品开发从一种“语言魔法”拉回了工程问题。

也就是说，真正的关键不再是：

- 有没有某个神秘 prompt 模板；
- 有没有一句能瞬间提升 40% 效果的咒语；

而是：

- 你的需求有没有结构化；
- 你的视觉参考够不够具体；
- 你的数据是不是独立、真实、可替换；
- 你的工具链是不是能补足模型本身的盲点。

说白了，**context engineering 不是一个花哨名词，而是把 AI 原型开发从玄学拉回方法论。**

## 三、Ravi 的三层框架：Functional / Visual / Data

这期视频里最有操作性的部分，是 Ravi 给出的三层上下文模型：

1. **Functional context**
2. **Visual context**
3. **Data context**

我觉得这套拆法非常好，因为它几乎天然对应了一个产品原型从“能说清”到“能看懂”再到“像真的”这三个阶段。

### 1）Functional context：先把产品要做什么讲清楚

Functional context 解决的是“这个东西应该怎么工作”。

它通常包括：

- 目标功能描述
- 关键交互逻辑
- 用户要完成的任务
- 约束条件
- 页面模块和行为要求

很多人以为这就是 prompt 本身。其实只对了一半。

Ravi 的意思不是“prompt 不重要”，而是说：

**那段文字说明很重要，但它只是其中一层。**

如果你只有功能描述，没有其它上下文，模型可能能生成一个大致能运行的东西，但很难精致，也很难稳定。

它更像一份“口头 briefing”，而不是完整的生产输入。

### 2）Visual context：告诉模型你想要什么样的界面语言

Ravi 特别强调 visual context，因为很多 AI 原型之所以看起来廉价，问题不是功能，而是视觉没有被约束。

视觉上下文可以来自：

- 现有产品截图
- 线框图
- Figma 参考
- 风格版（moodboard）
- 组件样式示例

这部分的价值非常直接：它能让模型少瞎猜。

你不告诉它“美感长什么样”，它就只能用训练里最平均、最模板化的审美去补全。于是你拿到的常常是一个“看起来像 AI 自动生成”的页面。

Ravi 的观点很对：

> wireframe 仍然是构建产品时非常重要的输入。

这也顺手打脸了一个常见误解：用了 AI，就不需要设计思考了。

恰恰相反。**设计没有消失，只是从“亲手画每一个像素”转向“给模型足够清晰的视觉意图”。**

### 3）Data context：让原型脱离假数据的塑料感

我认为这期视频里最被低估的亮点，就是 Ravi 对 data context 的强调。

他直接说：**JSON 是定义结构化数据的一个非常好方式。**

为什么这件事关键？因为很多 prototype 看起来像玩具，不是因为页面不够好看，而是因为底层数据是假的、扁的、临时拼的。

Ravi 演示时做的是一个音乐类产品。他没有只让 AI“做一个音乐页面”，而是先让模型生成一份独立的数据文件，包括：

- 流派名称
- 流派描述
- 代表音乐人
- 15 到 20 张关键专辑
- 每张专辑的发行时间、描述、标签等

然后把这份 JSON 单独作为输入喂给原型工具。

这个动作非常聪明，因为它带来三个好处：

#### 第一，数据和界面解耦

你可以不改界面逻辑，只替换 data.json，就把产品从 downtempo 切到 psychedelic rock。

#### 第二，原型更接近真实产品

当数据结构独立存在时，AI 生成的页面会更像真的在“消费数据”，而不是拿几段假文本硬塞进去。

#### 第三，后续迭代更容易

你要改的就不再是“整坨 prompt”，而是某个更清晰的层：

- 改功能 → 改 functional context
- 改风格 → 改 visual context
- 改内容 → 改 data context

这个结构感特别重要。它让 AI 原型开发第一次像软件开发，而不是像抽卡。

## 四、把三层合在一起，才叫 full-stack prompt

Ravi 后面展示了一个关键动作：把 functional、visual、data 三层合并成一个 **full-stack prompt**。

这里最值得注意的，不是“合并”本身，而是合并之前各层已经各自成形。

这意味着最终 prompt 不再是一个随手敲出来的大段说明，而更像一份编排过的输入包：

- 功能要求明确
- 视觉参考明确
- 数据文件明确
- 结构表达清楚

Ravi 还提到，他会用 Markdown 让这些信息更结构化、更容易被模型理解。

这背后的原则其实很朴素：

> **模型不是怕信息多，模型怕信息乱。**

很多人为了“简洁”，把所有要求压缩成几句话，结果是关键信息缺失。

真正更高级的做法，不是更短，而是更有组织。

所以 full-stack prompt 的重点，不是 prompt 本身变长了，而是你开始像组织一个系统输入，而不是像临场发挥一段指令。

## 五、为什么 MCP server 这段演示很重要

视频中还有一段非常能说明问题的演示：为了给音乐原型补齐专辑封面，Ravi 直接在 Claude Code 里做了一个自定义 MCP server。

原因也很现实：他找不到一个完全符合需求的现成服务，于是自己补了个工具层。

这个细节特别说明 context engineering 的边界其实比很多人想得更大。

很多人理解的上下文，只有“我给模型什么文字/图片”。

但 Ravi 这里实际上展示的是：

**工具本身也是上下文的一部分。**

因为模型是否能拿到：

- 真实封面图 URL
- 外部 API 结果
- 可计算或可查询的额外信息

会直接影响原型最终的真实感与完成度。

也就是说，context engineering 不只是整理已有材料，还包括：

- 判断缺什么
- 决定该不该接工具
- 用 MCP / API / scripts 把外部能力接进来

这就是为什么我觉得，未来做 AI 产品的人，多少都得懂一点“工具编排”。

你不一定要自己从零写 server，但你至少得有这种意识：

> 如果模型差一口气，那口气有时候不是 prompt，可能是工具。

## 六、好原型的目标，不是漂亮，而是能帮助决策

Ravi 讲了一个我特别喜欢的判断：

> A really good prototype is a good decision-making tool.

这句话太重要了。

很多团队现在做 prototype，依然停留在“做一个能展示的东西”。

但 Ravi 把标准抬到了更实用的位置：

- 这个原型是不是能帮你和用户讨论？
- 是不是能帮助团队判断方向？
- 能不能支持你验证某个关键假设？
- 能不能让用户进入“真实使用感受”，而不是停留在抽象评价？

这也是为什么他说，如果一个 prototype 不够真实，用户就会从“我会怎么用它”切回“我猜我可能会怎么想它”。

一旦进入后者，你拿到的反馈质量就会迅速下降。

所以判断 AI prototype 好不好，不能只看“做得快不快”。

更应该看：

1. 它是否足够真实，能引发有效反馈；
2. 它是否结构清晰，能持续迭代；
3. 它是否围绕一个明确决策问题来构建。

这是比“会不会出图、会不会生页面”高一层的产品判断。

## 七、PRD 没死，但线性流程该死一死了

视频后半段聊到 PRD，那段也很值得产品经理看。

Peter Yang 提到，大家以前在微软那种环境里，动不动就写 17 页 PRD。Ravi 的回应很成熟：

- PRD 不是完全没用；
- 只是过去那种严格线性的流程，不再天然成立了。

传统流程大概是：

1. 先 PRD
2. 再 wireframe
3. 再详细设计
4. 再 MVP

这个顺序之所以长期存在，是因为每个工件的生产成本都很高，所以你必须谨慎按顺序推进。

但现在，prototype 的成本大幅下降，很多探索可以并行发生。

Ravi 的意思非常明确：

- 有时候 PRD 在 prototype 前面；
- 有时候 PRD 在 prototype 后面；
- 都可以。

关键不在顺序，而在于：

**你到底想回答什么问题。**

这个观点我完全同意。

今天更合理的产品工作流，不是把 AI 当成“自动写文档的人”，而是把 PRD、原型、用户反馈看成几个彼此咬合的探索工具。

你不是照祖宗流程走，而是在组织一个更快的学习循环。

## 八、我从这期视频里提炼出的 5 个实战原则

如果你是产品经理、设计师、独立开发者，或者正在用 AI 做内部工具，我觉得这 5 条最该带走。

### 原则 1：别再只写 prompt，开始整理 context

与其花 40 分钟抠一句提示词，不如先问自己：

- 功能上下文够不够完整？
- 有没有视觉参考？
- 数据是不是独立、真实、可替换？
- 工具缺口有没有补上？

很多“模型不行”的抱怨，本质上是输入工程太烂。

### 原则 2：把数据层独立出来

如果你的原型里还有大量硬编码假数据，那它很难真正像产品。

至少试着：

- 单独维护 JSON / CSV / schema
- 让 AI 消费结构化数据
- 把“内容变化”和“界面变化”拆开

这一条会立刻提高原型的真实感与可维护性。

### 原则 3：视觉不是装饰，而是约束

设计参考并不是锦上添花，它是避免模型乱猜的重要约束条件。

没视觉上下文，AI 往往会产出“平均但廉价”的界面。

### 原则 4：把 prototype 当成决策工具，不是炫技玩具

做原型前先问：

- 我要验证什么？
- 我想帮谁做什么决策？
- 用户看到这个会给出哪类反馈？

如果这个问题不清楚，再快的 prototype 也只是热闹。

### 原则 5：学一点工具编排，收益会很高

MCP、API、外部数据源，这些不只是工程师的事。

在 AI 产品时代，谁更会把模型和外部世界连起来，谁就更容易跨过“好像能用”和“真的能用”之间那道坎。

---

## 九、这期视频真正打中的，不是技巧，而是方法论升级

我觉得 Ravi 这期最好的地方，是它没有把 AI 原型开发讲成一种神秘手艺。

它没有说：

- 去背 20 个神 prompt；
- 去找某个秘密模板；
- 去赌哪个模型更灵。

它讲的是更扎实、也更可迁移的方法：

**把 AI 当成一个需要被提供完整工作上下文的系统。**

这会带来一个很大的思维升级：

- 从“我怎么提问”升级到“我怎么设计输入环境”；
- 从“我怎么让它生成页面”升级到“我怎么让它完成产品工作”；
- 从“我怎么临场写 prompt”升级到“我怎么搭建一个可迭代的上下文架构”。

这不是术，而是底层操作系统在更新。

---

## 最后一句

如果你只记住这期视频的一句话，我建议记住这个：

> **Prompt 是一句话，context 是整个工作台。**

前者能让模型开始动；后者才决定它能不能把事做像样。

而 Ravi 真正展示给大家看的，不是“如何写一个更好的提示词”，而是：

**如何让 AI 不再像一个会胡乱发挥的实习生，而更像一个被正确配置过的产品团队成员。**

这差别，挺大。
