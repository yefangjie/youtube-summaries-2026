# 如何构建 AI Agent 团队：角色分工、反馈机制与协作方式

> 来源：IBM Technology  
> 原视频：[Building a Team of AI Agents: Roles, Feedback, & Teamwork Explained](https://www.youtube.com/watch?v=kqj22mWIdjU)  
> 日期：2026-04-21  
> 形式：基于英文字幕整理的中文解读

## 一句话总结

这期视频的核心观点很清楚：**复杂任务不该指望单个大模型硬扛，真正稳定可扩展的 Agent 系统，往往更像一支分工明确、能互相校验的团队。**

## 为什么“单体 Agent”不够了

视频一开头就把问题说透了：AI Agent 要解决的，通常是仅靠预训练知识和一次回答搞不定的复杂任务。比如开发一个移动应用，流程并不是“用户提需求 → 模型一次性吐出完美结果”这么简单，而是要经历：

- 梳理需求
- 设计架构
- 编写代码
- 调用工具与外部服务
- 测试和纠错
- 最终向用户汇报结果

也就是说，**复杂任务天然带有链路长、依赖多、需要反馈回路**的特征。单个 Agent 可以很快，但很容易顾头不顾尾；而多角色协作，才更接近真实生产环境。

## 视频提出的 7 类关键角色

IBM Technology 用“组建团队”的比喻来解释 Agent 设计，这个比喻非常准确，也很实用。视频中重点提到了以下几类角色。

### 1. Doer：执行者

执行者负责真正“把事做出来”。在人类团队里，它像初级工程师或具体执行人员；在 Agent 系统里，它可能负责：

- 写代码
- 生成文档
- 完成局部任务
- 执行被拆解后的步骤

它的特点是**擅长完成明确子任务，但不一定适合独自掌控全局**。

### 2. Planner：规划者

规划者的职责，是把复杂问题拆成可执行的小步骤。视频举了移动应用开发的例子：先规划用户需求，再规划应用架构，最后才进入编码。

一个好的 Planner，至少要做到三件事：

- 把大任务拆小
- 判断每一步需要什么技能和知识
- 产出一份清晰、可执行的计划

这其实就是许多 Agent 框架里最值钱的一层：**不是会不会回答，而是会不会组织任务。**

### 3. Tool Operator：工具操作者

这类角色负责和 API、脚本、数据库、Web 服务等外部工具打交道。它不是“懂很多”，而是“会正确调用”。

比如它要做的事情包括：

- 构造结构化工具输入
- 传对参数
- 读取返回结果
- 将结果交还给其他角色

很多 Agent 系统看起来聪明，本质上其实是**工具调用能力被包装得足够顺滑**。

### 4. Learner：学习者 / 检索者

视频把这个角色类比成团队里那个喜欢读资料、追踪市场变化的人。它负责从外部世界拿回信息，例如：

- 检索竞品信息
- 阅读博客和社交媒体
- 归纳用户偏好
- 从知识库中拉取上下文

这就是典型的 RAG 或外部检索层。重点不是“抓更多信息”，而是**抓相关信息，并把它送回正确的决策节点。**

### 5. Critic：批评者 / 反馈者

这类角色的重要性经常被低估，但视频讲得很对：**没有批评者的 Agent，只是在更快地产生错误。**

Critic 可以做的事包括：

- 检查回复是否幻觉
- 评估多个候选答案
- 给代码写 QA 测试并执行
- 对输出打分并挑选最佳版本

它相当于系统内部的“质检”和“逆向思考”机制。

### 6. Supervisor：监督者

监督者关心的不是某一步做得漂不漂亮，而是**整个流程有没有卡住、偏航或失控**。它可以在任务级、角色级、甚至项目级工作。

例如：

- 发现某个子 Agent 长时间卡死
- 判断失败发生在哪一步
- 协调不同角色重新分工
- 控制任务推进节奏

这层设计对长链路 Agent 特别关键。没有监督，系统常常不是“不聪明”，而是“没人收场”。

### 7. Presenter：呈现者

最后，系统总得有人把结果讲明白。Presenter 的职责，是把各个角色的成果整合成用户能理解、能使用、能决策的输出。

它不是简单复述，而是要完成：

- 汇总信息
- 解释结果
- 向用户说明系统实际做了什么
- 把复杂过程转成清晰结论

说白了，**没有 Presenter，再强的系统也可能在最后一公里翻车。**

## 这些角色如何组合起来

视频还提到一个重要观察：有些流行 Agent 模式，其实就是这些角色的组合体。比如 ReAct 模式，本质上可以拆成：

- **Reasoning**：规划者（Planner）
- **Action**：工具操作者（Tool Operator）
- **Observe**：反馈者（Critic / Feedback）
- **Answer**：呈现者（Presenter）

这说明很多“框架创新”，本质上并不是凭空造概念，而是在把团队分工结构化。

## 如何让每个角色“更擅长自己的工作”

视频后半段给出了四种强化角色能力的方法，我觉得这部分特别有价值，因为它不再停留在“角色划分”，而是开始讨论“岗位培训”。

### 1. Prompting：提示词设计

最直接的方法就是写清楚指令。对 Agent 来说，提示词就是岗位说明书、操作手册和行为规范。

例如：

- 卡住时先重试
- 不要越权修改计划
- 输出必须遵守某种格式
- 调用工具前先检查参数

一句话：**很多 Agent 的不稳定，不是模型太笨，而是岗位说明书太烂。**

### 2. Model Selection：模型选型

不同角色可能适合不同模型。规划者可能更适合推理型模型，执行者可能更适合便宜高效的通用模型，检索角色可能更依赖上下文处理能力。

视频强调，选模型时要看：

- 模型擅长什么
- 模型大小是否合适
- 是否需要 reasoning / thinking 能力
- 模型的人设或输出风格是否匹配角色

这点非常现实：**别拿贵模型干流水线体力活，也别拿便宜模型扮总架构师。**

### 3. Model Tuning：模型调优

如果任务非常关键，也可以通过微调或再训练，让某个角色更稳定地满足岗位要求。但这需要：

- 高质量正负样本
- 人工标注投入
- 额外算力成本

所以它更适合那些已经验证过价值、值得长期优化的流程，而不是一上来就乱砸资源。

### 4. Context：上下文配置

这是我非常认同的一点。视频把上下文比喻成“新同事入职时你给他开哪些系统权限、哪些文件、哪些数据库访问能力”。

上下文给得太少，角色无法完成任务；给得太多，又会形成噪音和干扰。真正有效的做法是：

- 给角色恰到好处的资料
- 让它知道哪些信息与当前任务相关
- 避免无关背景淹没判断

这其实是 Agent 设计里最容易被忽略、但最影响效果的部分之一。

## 对实际 Agent 系统设计的启发

这期视频最大的价值，是把“Agent 设计”从抽象概念拉回了工程现实。它提醒我们：

1. **复杂任务先拆角色，再谈提示词。**
2. **角色不是越多越好，而是要职责清晰。**
3. **反馈与监督不是可选项，而是稳定性的来源。**
4. **模型、工具、上下文要按岗位匹配，而不是一把梭。**

如果你正在做企业内部 Agent、自动化工作流、代码代理或者 AI 产品编排，这个框架很有参考意义。它足够简单，但不幼稚；足够工程化，也不会陷入空洞术语。

## 我的看法

我挺认同这期视频的底层判断：**未来更有竞争力的，不是“单模型更强”，而是“系统编排更成熟”。**

模型当然重要，但当任务开始跨步骤、跨工具、跨知识域，系统结构就会比单次回答能力更决定最终质量。很多人现在还在迷恋“一个超级 Agent 包打天下”，这思路短期能演示，长期很容易撞墙。

更靠谱的路线，反而像视频里说的：先想清楚团队里需要哪些角色，再决定谁负责计划、谁负责执行、谁负责质检、谁负责对外表达。这样做出来的 Agent，才更像一个能交付工作的系统，而不只是一个会说话的模型壳子。

## 结语

如果你最近在研究多 Agent、RAG、工作流编排或 Agent 工程，这期 IBM Technology 的视频值得一看。它没有搞那种唬人的大词，而是把问题讲得很朴素：**复杂任务，需要团队；团队要分工；分工之后，还要训练、约束和监督。**

这套逻辑，放在人类组织里成立，放在 AI Agent 里同样成立。

## 可直接复用的 Agent 团队设计清单

- 是否明确区分了规划、执行、反馈、监督、呈现？
- 每个角色是否有清晰的输入、输出和边界？
- 是否给关键角色配置了合适的模型？
- 是否为检索与工具调用设计了稳定接口？
- 是否有内部 critic / QA / reranking 机制？
- 是否有 supervisor 负责兜底和故障定位？
- 最终输出是否经过 presenter 重新组织，而不是原样拼接？

如果这些问题里有一半你答不上来，那你的 Agent 系统大概率还停留在“能跑”，离“能稳定交付”还有点距离。
